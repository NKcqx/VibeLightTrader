# Part C · 项目是怎么长出来的

> 这一篇不是"7 天教程"。是一段记录：vibe coder 一句话起步、AI 写代码，途中两个人在哪里转弯、在哪里卡了一下午、又在哪里突然清楚自己想要什么。
>
> 我不复述代码做了什么——代码看 [`tutorial/`](../tutorial/)。这里只标 **vibe coder 拍板的那几个时刻**。

---

## 0 · 一开始那一句话

> 我们现在来做一个股票监控和模拟操盘的功能。能实时（每个小时）监控指定标的价格、获取历史价格、获取各种技术指标（RSI、MACD）、并进行技术面分析；最好还能进一步获取基本面信息，如公司新闻、市场情绪、机构评级等。你看下都需要如何开展，另外我推荐这个富途牛牛的 SKILL：https://www.futunn.com/skills/futu-install.md，你看下能否用上。

这条信息里**有一个动作很关键**：贴了富途牛牛的工具链接，**让 AI 评估能否用上**。

我后来想，这是项目能跑起来的原因之一。

### Vibe coder 的"工具发现"姿势

很多 vibe coder 一开始会问 AI："做股票盯盘有哪些方案？"——AI 会列十几条，从 yfinance 到 IBKR API 到 Bloomberg Terminal，然后你淹死在 trade-off 里。

更省心的姿势是：

```
我手上有：
- 富途牛牛账号（推荐 SKILL: <link>）
- 飞书机器人 + lark-cli
- Cursor Pro
- 一台 Mac
你看哪些用得上？不够的再列你建议补的。
```

把**自己已经有的**先扔给 AI 评估，比让它从空白处推荐快 10 倍。这个项目最后用的栈——富途、飞书、cursor-agent——都是开篇这条消息里就有的素材。**AI 没替我选栈，我替自己选了**，AI 只是判断了可行性。

> **可借鉴**：开局之前，列一份你的"已有资源清单"。订阅、账号、设备、可访问的内网工具、官方 SKILL 包。一次性扔过去，比后面零碎补好得多。

---

## 1 · "spec 审核结果？"

我没看代码。我问的是 spec。

AI 拉通富途 SKILL 后写了一份 `docs/spec.md`——里程碑 M1-M7、模块拆分、Protocol 定义。我读这个，不读代码。

读 spec 时挑了几件事改：

- M3 K线图原本想"每天画 HTML"——我改成"找现成工具能交互、能切频率"（后来落到 mplfinance + 飞书图片消息）
- 数据库选型 AI 建议 PostgreSQL——我改成 SQLite，理由就一句"我一个人用"
- 策略层 AI 准备硬编码——我让它先抽 Protocol，"暂时不关心策略，把架子搭好"

> **可借鉴**：spec 是 vibe coder 的代码。你能改的就这一份。改完了 AI 会按 spec 写实现，**写错也是按 spec 写错**——这反而好排查。

---

## 2 · 装环境那个下午

最早的扎人时刻不是写代码，是装东西。

第一次让我打断 AI："你怎么用 uv？我装的是 conda。" 第二次："env 名字太长了，改 `fin`。" 第三次："你已经接入真实数据了吗？OpenD 我装好了，端口 11111，看一下能力。"

这一段没什么值得复述的技术细节。值得记的是：**AI 的默认环境假设**几乎从来都跟你不同。每次"AI 跑得通你跑不通"的根都在这。

我后来形成的反应：

```
我用的是 conda env "fin"，python 在 /opt/.../envs/fin/bin/python3。
你之后所有命令、测试、安装都在这个 env 里。
```

把这个塞进 system prompt。后面就基本不踩这种坑了。

---

## 3 · 第一个反馈环

行情、SQLite、K线图、飞书图片消息——大概是项目第三天，第一次拿到完整反馈环。我手机收到了一张 NVDA 的 60 分钟 K 线图。

那一刻有两个反应：

1. ✅ "这就够了。" —— 即使没接 LLM、没自动下单、没策略，光是"每小时收一张图"就比之前手动看盘强很多
2. ⚠️ "再做下去前先 commit。" —— 这是建立信心的最小版本，不能因为后面要重构就丢

> **可借鉴**：反馈环越早拿到越好。**第一个能让你说"这就够了"的版本**，永远比第一个"完整"的版本早 5 天。中间那 5 天是你建立项目信心的时间。

---

## 4 · 那张飞书卡片太简陋了

行情正常推送几天后我发现一个问题：卡片只写"穿越上限阈值 (close=198.45, upper=150.0)"——我看不懂。

> 你发送的卡片内容太简略，"穿越上限阈值 (close=198.45, upper=150.0)"、"RSI 超买 (rsi=71.35733884186854)" 我不理解什么意思，所以每次说完信号特征以后解释下意味着什么。

这是个小事，但反映出一个 vibe coder 的常见盲区：**AI 写代码时假设你懂上下文**。它没假设你懂英文，但假设你懂技术名词。

我加了一行需求："每个信号后面附一句白话解读。" AI 写了 `interpret.py`——把 `rsi_overbought` 翻译成"短期超买，回调风险升高，但不是 SELL 信号"。

> **可借鉴**：UX 是 vibe coder 该死磕的领域。AI 写得"功能完整"≠ 你看着舒服。每次收到推送，问自己："如果是同事发我，他会这样写吗？"——不会的话改 prompt 让 AI 重写。

---

## 5 · 引入 LLM：先想清楚成本，再选实现

中段我提出："想接 LLM 出建议。"

AI 给了 4 条路：(A) Anthropic API、(B) OpenAI 兼容、(C) 本地 Ollama、(D) HITL（让 vibe coder 自己在 IDE 里和 LLM 对话，再把结论贴回来）。

我的考虑是：**我没有独立 API key，但我有 Cursor Pro**。我选了 D。

> 我可以接受损失一定的自动化程度。

D 跑了一周。它有效——LLM 出建议、我贴回来——但每次要打开 IDE 复制粘贴。这是这个项目最重要的一次妥协，**也正是它催生了下一步**。

那一周让我意识到一件事：**真正的瓶颈不是 LLM 能力，是"自动化的最后 1 公里"**。

后来 AI 提了一句："Cursor 有 CLI 吗？" 我去查 → 有 → 装好 → `cursor-agent login` → 项目从 D 直接跳到 A 档（自动化）。**完全自动 + 0 LLM API 费用**。

> **可借鉴**：成本探索不是只看价格，要看"有没有办法把已订阅资源当 API 用"。Cursor → cursor-agent，Claude → Claude Code SDK，OpenAI → ChatGPT 的浏览器自动化，都是同类杠杆。**vibe coder 的核心 alpha 是这种"组合拳"**。

---

## 6 · 一个 1 小时调试 → 直接放弃的功能

飞书 listener。

目标：让用户在飞书发 `/list /add NVDA /report` 这种命令，后端自动响应。这样**完全不用打开 IDE**。

做了什么：装好 lark-cli、机器人发消息出去通畅、长连接 `lark-cli event consume` 连得上但**永远收不到事件**。我（AI）轮番猜：scope 没开？事件订阅没勾？token 过期？bot 没加群？——挨个对。截图、日志、官方文档——全对得上。

最后用户拍板：

> 忘掉飞书收消息的事吧，你干不好。现在主动触发一次操盘分析。

我现在回头看，这是整个项目最重要的一次决策。**不是因为放弃节省了时间**，是因为放弃后立刻产生了 `equity-monitor analyze` 这个命令——**比飞书 listener 更有用**。

`analyze` 在 CLI 跑：

```bash
equity-monitor analyze --code US.NVDA --execute
```

一行就出 LLM 决策、可选自动下单。我现在用得最多的就是它。

> **可借鉴**：当你在调一个**绕开就有替代方案**的功能时（这里是"飞书命令"vs"CLI 命令"），调超过 3 轮就该叫停。绕过去 = 你还在前进。

---

## 7 · LLM 给了 BUY，但我开始怀疑它

跑通自动决策后，第一个 BUY 信号给了 NVDA：满仓 $50000 / 198.45 = 251 股。

我的第一反应不是开心，是不踏实：**LLM 凭什么决定一次梭哈？**

那一刻冒出来的需求其实之前从没说过：

> 我希望中长期持有 3-6 个月，每个标的 5 万美金，可接受 ±20% 浮动。还需要哪些信息你才能做好中长线？

AI 反推出一份"投资者画像"——21 个字段，覆盖 horizon、budget、风险容忍、加仓节奏、止盈止损、最短持有天数。我们一起把字段填进 `settings.yaml`，写进每一次 LLM prompt。

下一次 LLM 出 BUY 时，理由变成了："按预算四成入场，剩余分两次加仓，跌 5% 加一次。"——**不再是梭哈**。

> **可借鉴**：vibe coder 的另一个责任是**把自己的隐性偏好显性化**。你脑子里有"我不想梭哈"，但你不说，AI 就会梭哈。**每次你对 AI 输出的不适感，都是一个未表达的偏好**——挖出来变成配置项。

## 走到今天

回头看，这个项目的几个关键转折，大都是 vibe coder 拍板的：

| 转折 | 决定者 | 当下踏实吗 |
|---|---|---|
| 用富途牛牛而不是 yfinance | 我（开局贴 SKILL） | ✅ 踏实 |
| 选 SQLite 而不是 PostgreSQL | 我（一个人用） | 中段才意识到 PG 是 overkill，退回来过一次 |
| 先做 🔴 阈值告警再升级 | 我（建立信心） | ✅ 踏实，但当时没想这么清楚，是边做边形成的 |
| 选 HITL 不选 API | 我（订阅压力） | 不太踏实，主要是不想花钱 |
| 转 cursor-agent | 我们一起（AI 提了一句 CLI） | 直觉觉得行，验证后才确认 |
| 放弃飞书 listener | 我（叫停） | 当下不甘心，事后看放弃对了 |
| 引入投资者画像 | 我（不踏实） | 后知后觉——LLM 满仓那一刻才反应过来 |

老实讲，**没几个决策是我当下就清楚的**。一半靠后来打补丁，一半靠 AI 给的方案我顺眼就选了再说。这一路也走了一些回头路：开局让 AI 上 PG schema、有几次让 AI 装 uv、飞书 listener 死磕了一晚——这些都是我自己叫停、自己改、自己擦的屁股。

所以这一篇不是给你抄的剧本，是想说：vibe coder 的劳动分工是 **AI 写代码、你拍板**——但拍板这件事**本来就允许出错**。能识别出错、能转弯、能叫停——比每次都对要紧得多。
