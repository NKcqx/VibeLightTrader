# Part B · 与 AI 协作的 7 个 meta-skill

> 这是这份指南的核心。代码 AI 都会写，**但 AI 不会管理你跟 AI 自己的协作过程**——那是你的活。
>
> 7 个 skill，每个都给：① 一句话定义 ② 为什么重要 ③ 怎么具体说（话术 / prompt 例子）④ 反例。

---

## Skill 1: 先要 plan 再动手

**一句话**：动手前先要 plan，看完 plan 再说"go"。

**为什么**：AI 一拿到任务就改代码 = 没经过你脑子的修改 = 你 review 全靠运气、commit 难拆、出问题难回滚。

**怎么说**：

```text
先告诉我 plan，不要直接动手。
把这个需求拆成 3-5 个 commit 的粒度，每个 commit 我能 review 完。
Plan 通过我再说 go。
```

**反例 vs 正例**：

> ❌ "帮我加个止损功能" → AI 直接改 5 个文件 + 写测试 + commit + push
>
> ✅ "我想加止损功能。先告诉我：(1) 会改哪些文件 (2) 跟现有 strategy 怎么衔接 (3) 是否要加配置项 (4) 测试怎么写。看完 plan 我决定 ok 才动手。"

**何时跳过**：tiny task（改个数字、改个字符串、加个 print）。task 越复杂越要 plan。

## Skill 2: todo list 是方向盘

**一句话**：复杂任务要让 AI 维护 todo 列表，逐项 in_progress / completed。

**为什么**：

- AI 的"短期记忆"会被自己的输出污染——做完一项很容易忘下一项
- todo 列表是你**实时**看到 AI 在做什么的窗口
- 任务卡住时，看 todo 立刻知道断在哪儿

**怎么说**：

```text
把刚才聊的 4 件事整理成 todo list，逐项 in_progress / completed。
开始干第一项。
```

```text
todo 还有哪几项没干完？给我一份当前状态。
```

**反例 vs 正例**：

> ❌ "做完 A 顺便把 B 也做了" → AI 做了 A 就忘了 B
>
> ✅ "todo 1 = A, todo 2 = B, todo 3 = C" → AI 完成 A 后会接 B

**这个项目的实践**：M4-M7 阶段每个里程碑都让 AI 维护 todo，每完成一项 mark 为 completed。这是我跟 AI 的"心电图"——一看 todo 卡住，就知道要介入。

## Skill 3: subagent-driven 工作流

**一句话**：把可以并行的任务分给多个 subagent，单线程跑 1 小时不如并行跑 10 分钟。

**为什么**：vibe coder 的瓶颈不是"AI 算力"，是**等待时间**。能并行就不要串行。

**怎么说**：

```text
subagent driven 模式跑这个任务。
```

```text
把这两件事拆成 2 个 subagent 并行做：
- agent 1：调研 X 的实现方式
- agent 2：调研 Y 的实现方式
```

**适合的场景**：

- ✅ 探索代码库（一个 agent 找 X，另一个找 Y）
- ✅ 写多个独立模块（一个 agent 写 A 模块，另一个写 B 模块测试）
- ✅ 调研 trade-off（一个 agent 调研方案 1，另一个调研方案 2）

**不适合的场景**：

- ❌ 任务有强依赖（B 必须在 A 完成后才能开始）
- ❌ 需要共享状态（两个 agent 改同一个文件 = 冲突）
- ❌ 需要持续协商决策的任务（subagent 看不到主对话上下文）

**这个项目的实践**：M5-M7 阶段用 subagent 并行跑 4 个任务，比单线程快约 60%。journal 模块（writer / metrics / errors / tests）就是 4 个 subagent 同时干的。

## Skill 4: 看到跑偏立刻打断

**一句话**：AI 跑偏 30 秒就要叫停，越晚成本越高。

**为什么**：vibe coder 最重要的肌肉记忆。AI 在错误方向上写代码 = **你后面要花同等时间清理**。

**5 个常见跑偏信号 + 真实例子**：

| 信号 | 这个项目的真实对话（已脱敏） |
|---|---|
| ① 用错环境 | "你怎么又用到 uv 了？我不是说要复用 conda 'fin' 环境吗" |
| ② 过度工程 | "我暂时不关心策略，我只想先把架子搭好" |
| ③ 钻牛角尖 | "这个功能你已经做了很久了，暂时放弃吧" |
| ④ 没装好就开干 | "等等，OpenD 装好了吗？开盘了吗？没开盘你跑也观察不了吧" |
| ⑤ 假设你有它没有的信息 | "你看不到我的飞书后台，我截图给你" |

**怎么打断**（不要客气）：

```text
暂停一下
等等
停
```

AI 不会受伤。

**真实场景**（**这一刻打断省了 30 分钟**）：

> AI 已经开始 `pip install uv`、改 pyproject.toml……我打："暂停一下，你怎么又用到 uv 了？我不是说这个项目用 'fin' conda 环境吗。"
>
> AI："抱歉，已停手。让我撤回 uv 相关改动，改回用 conda 'fin' env。"
>
> 如果晚 5 分钟打断，已经 commit 了 pyproject.toml + uv.lock，要回滚一堆。

## Skill 5: 让 AI 用原始材料诊断

**一句话**：AI 想猜原因时，先让它看原始素材。

**为什么**：AI 没看过你的截图、没跑过你的命令、没看过你飞书后台 ——它"觉得应该是 X"经常是错的。

**怎么说**：

```text
贴 lark-cli auth status 的原始 JSON 输出再判断。
```

```text
我截图给你，你照着图看。
```

```text
运行 X，把 stderr 完整粘回来，不要省略。
```

**反例 vs 正例**：

> ❌ "AI: 应该是 scope 配置错了" → 你跑去飞书后台改半天 → 改完还不行 → AI 又猜下一个原因
>
> ✅ "AI: 你先 `lark-cli auth scopes --format json` 给我看" → 看完才发现 user 视角 ≠ bot 视角，问题根本不在 scope

### 真实案例（飞书 listener 调试）

> 我（AI）猜了 3 次"是 scope 没开"，每次让用户改飞书后台。改完还是收不到事件。
>
> 最后让用户跑 `lark-cli auth scopes --format json`，发现输出里 `"tokenType": "user"`——我之前一直在用 user 视角的 scope 列表去判断 bot 应该有的 scope，**根本对不上**。
>
> 真正的解法是另一回事，但**让我节省了 1 小时**的关键就是看到那行 JSON。

**心法**：每次 AI 说"应该是 X"——你都问一句"**根据什么？**"。AI 答不上来，就让它先取证。

## Skill 6: 环境差异化沟通

**一句话**："你跑得通我跑不通"——把环境差异显式告诉 AI。

**为什么**：AI 在它沙盒里跑通的命令，你本机环境可能：

- 路径不一样（cwd / 工作目录 / `~/Documents/Code/foo` vs `~/foo`）
- 权限不一样（sudo / sandbox / 公司管控）
- 依赖装在另一个 conda env
- 网络不一样（公司内网代理 vs 公网）
- 平台不一样（macOS vs Linux，Apple Silicon vs Intel）

**怎么说**：

```text
我跑同样的命令报错: <粘贴完整错误>
我的 cwd 是 ~/Documents/Code/equity-monitor
我用的是 conda env 'fin'，python 是 /opt/.../envs/fin/bin/python3
```

**真实案例**（这个项目）：

> 用户：`equity-monitor chart US.AAPL --push` 报错 `lark-cli`：
>
> `--image: --file must be a relative path within the current directory, got "/Users/.../var/snapshots/US_AAPL_60m_20260503_064821.png"`
>
> AI 之前测试时是相对路径调用，没想到用户在不同 cwd 下调命令时变成了绝对路径。**只有用户提供完整 traceback** 才能定位——是 lark-cli 不允许绝对路径。
>
> 修：在 `send_image` 里把绝对路径转成相对 cwd 的相对路径。

**心法**：报错时**不要省略**。完整 traceback、完整命令、完整 cwd——一次性给齐，省 3 轮往返。

## Skill 7: 何时叫停一个功能

**一句话**：会叫停的 vibe coder 比死磕的 vibe coder 走得远。

**为什么**：项目里**总有一两件 AI 干不好的事**——不是你的错，也不是 AI 的错，可能是工具/平台/环境的某种 misalignment。会识别这种事并绕开，是最高级的 vibe coder skill。

**叫停信号**：

- ⚠️ 同一个 bug 调 **超过 3 轮** 还没进展
- ⚠️ AI 开始重复说"再试这个看看"——重复 = 它也没主意了
- ⚠️ 你已经在改**非项目代码**（飞书后台、防火墙、系统包、IDE 设置）
- ⚠️ 修一个 bug 引出 3 个新 bug
- ⚠️ 已经超过原计划时间 2-3 倍

**怎么说**：

```text
这个先放一放，下个版本再搞。
忘掉 X，做 Y 吧。
```

### 这个项目的真实负面案例：飞书 listener

**目标**：让飞书机器人接收用户消息，触发 `/list /add /report` 等命令——实现"完全靠飞书交互、不打开 IDE"。

**做了什么**：

1. ✅ 装好 lark-cli，机器人能发消息（出去通畅）
2. ❌ `lark-cli event consume` 长连接连上了，但永远收不到事件
3. 🔁 调试 5 轮：刷 token、查 scope、查事件订阅、启 polling fallback、对照官方文档……
4. ❌ 飞书后台显示一切正常，但事件就是不来

**我（AI）的诊断结论**：可能是飞书 Open Platform 端某个隐性配置（事件订阅域名 / bot 加群配置 / 用户级 vs 机器人级权限），但仅靠 CLI 工具看不见。

**用户的最终决定**：

> "忘掉飞书收消息的事吧，你干不好。现在主动触发一次操盘分析。"

**这是对的决策**。继续修可能再花 2-3 小时；放弃后立刻转向"主动 analyze 命令"，半小时拿到下一个里程碑——还顺便催生了 `equity-monitor analyze` 这个**比飞书 listener 更有用**的功能。

**复盘心得**：

| ✅ 做对了什么 | ❌ 做错了什么 |
|---|---|
| 用户敢拍板"放弃" | 我（AI）应该早 1-2 轮主动建议"绕过去" |
| 转向新方向后立刻有产出 | 用户陷入了"已经投入这么多不甘心放弃"的沉没成本 |
| 没写没调通的代码 push 到 main | 调试过程消耗了一整个晚上 |

**心法**：项目里出现"硬钉子"（看不见的平台配置 / 上游 bug / 工具不配合）时，**绕过去比怼上去聪明**。

## 元元 skill：心理建设

最后一条，也是隐形的：**vibe coder 要承受一种新的"焦虑形态"**。

- 你看到 AI 在写代码 → 不像传统编程，你**插不上手**
- 你不太懂代码 → 但你又必须**做决定**
- 出了问题 → 你**不知道**是 AI 的锅还是你的锅

**心法**：

- ✅ 把"插不上手"换成"在做产品决策"——你的角色变了，不是变弱了
- ✅ 不懂代码没关系，**懂业务**和**懂资源**就够了
- ✅ 出问题就分两种：**让 AI 解释**或**叫停**——不要自己硬上
