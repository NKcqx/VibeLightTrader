# Part A · 立项与心智设定

> 你写下"我想做个量化盯盘"的那一刻，已经走错了第一步。这一节告诉你怎么走对。

---

## 1. 你不是在做"量化系统"，你在做"一个解决你具体问题的工具"

vibe coder 最容易栽的第一跤：把"做一个量化系统"当成项目目标。这是大词。AI 听到大词只会给你一个大而无当的脚手架——MVC 三层 + 多策略框架 + 回测引擎 + 风控模块……跑起来你才发现没一项是你真要的。

**先用 1 句话回答 5 个具体问题**：

1. 我有多少钱要管？
2. 我多久看一次盘？
3. 我能接受多大回撤？
4. 我有几个标的，固定的还是要选？
5. 我希望 AI 帮我做多少决定？

把答案合并成一个**项目宪法**。这个项目当时的版本是：

> **5 万美金 / 标的，3-6 个月持有，可接受 ±20% 浮动，标的固定 NVDA + MSFT，AI 替我盯盘 + 出 BUY/SELL/HOLD 建议，最终下单到富途模拟账户。**

这一句话是项目所有后续决策的锚点。后面 AI 跑偏、你纠结要不要加新功能、新冒出来的"好主意"该不该接受——都从这一句话推。

✅ **写不出来 1 句话，就还不该开始**。

## 2. 资源盘点：vibe coder 的"原料清单"

vibe coder 不写代码，但你**要管理资源**——这跟做产品经理是一样的。盘点你已经拥有的：

| 类型 | 选项 | 这个项目用了什么 |
|---|---|---|
| IDE / coding agent | Cursor / Codex / Claude Code / Aider | Cursor + cursor-agent CLI |
| LLM 订阅 | Cursor Pro/Max / Claude Pro / OpenAI Plus / 自有 API | Cursor Pro（Opus 4.7） |
| 数据 | 富途 / Yahoo / Bloomberg / IBKR | 富途 OpenD（免费） |
| 通知 | 飞书 / 邮件 / Slack / 微信 | 飞书 + lark-cli |
| 持仓追踪 | 自建 DB / broker 模拟账户 | 富途模拟账户（外部可见关键） |
| 时间预算 | 每天 1h / 每周末 / 整周 | 工作日间隙 + 周末整块 |
| 风险偏好 | 保守 / 平衡 / 进取 | 平衡（20% 回撤） |

### 关键洞察：**复用你已有的订阅和账户**

这个项目省下了：

- **LLM API 费用** —— 用 cursor-agent 调用 Cursor 订阅的 Opus，不用自掏 API key（本来一个月 $50-200）
- **量化数据费用** —— 用富途免费 API，不用买 Bloomberg / Polygon（本来一个月 $50-300）
- **持仓追踪 UI** —— 富途 App 自带模拟账户视图，不用自己写持仓页面（本来要 2-3 天）

这三项加起来每月 $100-500 + 少写一周代码。**vibe coder 的第一个杠杆**：把订阅当 API 用。

✅ **盘点写下来，让你的 AI 也看到。**之后讨论"这个功能用什么实现"时，AI 会优先在你已有的资源里挑。

## 3. 决定 AI 替你做多少：三档自动化光谱

你要先决定，最终形态长什么样：

| 档位 | 描述 | 适合谁 |
|---|---|---|
| 🟢 **全自动** | 定时调 LLM → 自动下单到模拟账户 → 飞书播报 → 你只管收消息 | 信任 LLM、不想盯盘、风险可控（小金额）|
| 🟡 **半自动** | LLM 出建议 → 飞书卡片推给你 → 你点确认才下单 | 想保留最后一步控制权 |
| 🔴 **单点工具** | 不接 LLM，只是 K线推送 + 阈值告警，你自己手动决策 | 不信任 LLM、想自己做研究 |

这个项目最终落在 🟢，但**不是一上来就 🟢**——是先 🔴（阈值告警），再 🟡（HITL：人在环），最后 🟢（cursor-agent 接管）。

> **vibe coder 心法**：不要一上来就要全自动。**先做 🔴 跑通，是建立信心的关键**。
>
> - 🔴 跑通 → 你确信数据流没问题
> - 🟡 跑通 → 你确信 LLM 的判断你看得懂
> - 🟢 跑通 → 你确信整条 pipeline 不会暴雷

**反例（真实警示）**：vibe coder 想一步到位 🟢，结果 LLM 判断不准、下错单、自己又看不到中间过程——这是 50% 失败项目的死法。

## 4. vibe coder 的核心责任：边界要画清

| 你做的 | AI 做的 |
|---|---|
| 决定要解决的真问题 | 把问题翻译成代码 |
| 决定技术方案的 trade-off | 列 trade-off 选项给你 |
| 决定何时叫停 | 警告"代价高"但**不会主动停** |
| 验证产出符合预期 | 自检代码错误（部分） |
| 做 UX 决策（卡片样式、推送频率） | 提供候选 |
| 知道你环境的特殊性 | 假设标准环境 |
| 看到你看到的东西（飞书后台、富途客户端、本机环境） | 全程**盲盒**推断 |
| 同意 / 否决新功能 | 永远倾向"可以加" |

### 反直觉点：**vibe coder 比工程师更要主动拍板**

很多人想象的 vibe coder：拎包入住，AI 全包。
真实的 vibe coder：你越被动，AI 越容易越界、越容易跑偏、项目越容易烂尾。

具体表现：

- AI 列了 3 个方案 → **你必须选一个**，不能让 AI 选
- AI 说"这里有 trade-off，X 是 OOXX，Y 是 ××××" → **你必须拍板**
- AI 觉得"既然要做就一起做了 Z" → **你必须问"Z 是必须的吗"**
- 测试报错 → **你必须把完整 traceback 贴回去**，不能让 AI 凭印象修

**核心心法**：AI 不是替你做决定的，是**替你执行决定**的。你的角色是产品经理 + 测试 + 验收 + 决策者，**不是"用户"**。

## 5. 心智设定 checklist（开干前问自己）

下面 5 个勾全打满，再 clone 一个空 repo。打不满就再想想。

- [ ] 我能用 **1 句话**说出这个项目要解决什么问题
- [ ] 我已经盘点了我的资源（订阅 / 账号 / 时间）
- [ ] 我决定了自动化光谱在哪一档（🟢/🟡/🔴），并且**接受先做 🔴 再升级**
- [ ] 我知道哪些事**只能我做**（看 [Part E](./E-ai-cant-do-this.md)）
- [ ] 我同意：AI 不替我做决定，是替我**执行**决定

## 6. 给你的 AI 看的"开局 prompt"模板

把下面这段贴给你的 AI，作为项目第一条 system message 或第一条对话——能让 AI 一开始就在正确的轨道上：

```
我们要做一个项目。开始前请遵守以下约定：

【项目宪法】
<在这里写你的 1 句话目标。例如："5 万美金 / 标的、3-6 个月持有、
2 个固定标的、AI 出建议 + 自动下模拟单、飞书播报">

【我已有的资源】
- IDE：Cursor，订阅 Pro
- LLM：通过 cursor-agent CLI 调用 Cursor 订阅
- 数据：富途 OpenD（本机已装）
- 通知：飞书机器人 + lark-cli
- 模拟账户：富途 SIMULATE（手机 App 可见）

【自动化光谱】
最终目标 🟢 全自动，但分阶段做：先 🔴 数据 + 告警，再 🟡 LLM 建议 +
人工确认，最后 🟢 自动下单。每个阶段都要能独立跑通。

【协作规则】
1. 任何"做 X 功能"的请求，先给 plan，我说 go 才动手
2. 任何超过 3 个文件的改动，先列 todo 列表
3. 我说"等等 / 暂停"立刻停手不要 push 你的修改
4. 报错让我贴 traceback，不要凭印象修
5. 我环境的特殊性（路径、conda env、平台后台）我会主动告诉你

明白请回复"明白，请说项目宪法"。
```
